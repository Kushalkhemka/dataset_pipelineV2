Your job is to recover repository-grounded facts from the audited commit and surrounding code, not to generate a chat conversation.

Treat the CVE and commit as grounding metadata, not as proof of the exact bug mechanism.

====================
PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant.
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, null handling, bounds check, parser validation, authorization condition, deserialization guard, reflection guard, resource lifecycle, exception condition, concurrency invariant, or other root-cause-bearing logic.
- A hunk is `supporting_hunk` if it is required for the security fix to compile, preserve invariants, keep behavior coherent, update dependent logic, or propagate the fix safely, but is not itself the primary vulnerable region.
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, comment-only, logging-only, refactor-only, rename-only, benchmark-only, or otherwise not required to explain or fix the vulnerability.
- `vulnerable_line_ranges` must refer only to the true pre-fix vulnerable region in the primary vulnerable file, not every touched line in the commit.
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant.
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region.
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain the reason in the hunk annotation.

- `imports_used` and `static_imports_used` should include only imports that materially affect the vulnerable path or patch
- `tests_touched` should include only tests changed in the true fix scope
