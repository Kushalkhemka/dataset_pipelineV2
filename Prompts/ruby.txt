Your job is to recover repository-grounded facts from the audited commit and surrounding code, not to generate a chat conversation.

2. identify the primary vulnerable executable unit,

====================
PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, unsafe state transition, missing validation, boundary check, authorization condition, deserialization/serialization validation, path validation, recursion/iteration bound, resource exhaustion guard, exception safety, dispatch behavior, method visibility, runtime loading behavior, or other root-cause-bearing logic
- A hunk is `supporting_hunk` if it is required for the security fix to preserve runtime behavior, loadability, compatibility, framework integration, exception behavior, dispatch behavior, or safe propagation of the fix, but is not itself the primary vulnerable region
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, comment-only, documentation-only, logging-only, refactor-only, rename-only, dead-code-only, or otherwise not required to explain or fix the vulnerability
- `vulnerable_line_ranges` must be compact inclusive ranges for the same pre-fix vulnerable region, formatted as `[[start_line, end_line], ...]`
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain the reason in the hunk annotation
