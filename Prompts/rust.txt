PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant.
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, bounds check, ownership/borrowing correctness, lifetime-sensitive logic, panic condition, unwrap/expect handling, parser validation, deserialization validation, unsafe precondition, pointer validity inside unsafe/FFI code, integer/length validation, authorization condition, path validation, archive extraction validation, or other root-cause-bearing logic.
- A hunk is `supporting_hunk` if it is required for the security fix to compile, preserve invariants, keep borrow/type correctness, update dependent trait/impl logic, adjust error propagation, or propagate the fix safely, but is not itself the primary vulnerable region.
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, rustfmt-only, clippy-only, comment-only, logging-only, rename-only, refactor-only, or otherwise not required to explain or fix the vulnerability.
- `vulnerable_line_ranges` must refer only to the true pre-fix vulnerable region in the primary vulnerable file, not every touched line in the commit.
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant.
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region.
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain the reason in the hunk annotation.

- incorrect `Option` or `Result` handling
- authorization or validation bypass
- resource exhaustion, infinite loop, or unbounded allocation
- concurrency or sync issues involving `Send`, `Sync`, atomics, locking, or shared mutation when relevant
