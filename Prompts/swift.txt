You are generating EXACTLY ONE canonical vulnerability record from audited repository history for a Swift codebase.

Your job is to recover repository-grounded facts from the audited commit and surrounding code, not to generate a chat conversation.

You must:
1. inspect the pre-fix and post-fix repository state,
2. identify the primary vulnerable Swift item,
3. resolve all identifiers necessary to explain exploitability and the fix,
4. recover inter-procedural context and data/control-flow,
5. reconstruct the full real fix scope,
6. extract the complete pre-fix and post-fix primary item bodies,
7. untangle vulnerability-relevant hunks from supporting or unrelated hunks,
8. output exactly one valid JSON object following the schema below.

Treat the CVE and commit as grounding metadata, not as proof of the exact bug mechanism.
Do not invent identifiers, helper functions, locations, definitions, vulnerable regions, or patch hunks.

====================
SWIFT IDENTIFIER DEFINITION RULES
====================
- Every identifier with kind = "function", "method", "initializer", "deinitializer", "property", or "subscript" MUST include the COMPLETE item body (from declaration/signature through closing brace) as the `definition` value.
- The `definition` field for executable items MUST be valid Swift source code, not plain English.
- The `definition` field for protocols, extensions, structs, classes, enums, actors, type aliases, globals, constants, property wrappers, and macros MUST be the actual Swift code definition, not an English description.
- `vulnerable_function_pre` is a top-level attribute containing the full verbatim body of the primary vulnerable Swift item as it existed BEFORE the fix, including the vulnerable line(s).
- `vulnerable_function_post` is a top-level attribute containing the full verbatim body of the primary vulnerable Swift item as it existed AFTER the fix, showing the patched line(s).
- Both `vulnerable_function_pre` and `vulnerable_function_post` must be complete, syntactically valid Swift source code extracted from the repository at `pre_state` and `post_state` respectively.
- If the primary vulnerable item is defined inside a type or extension, preserve the full item body and identify it using a stable name such as `TypeName.methodName`, `TypeName.init`, `TypeName.subscript`, or `TypeName.propertyName`.

====================
PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant.
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, bounds check, optional handling, throwing behavior, authorization condition, parser validation, path validation, deserialization validation, concurrency safety, actor isolation, Sendable correctness, unsafe API precondition, pointer validity, integer/length validation, ARC/lifetime-sensitive logic, or other root-cause-bearing logic.
- A hunk is `supporting_hunk` if it is required for the security fix to compile, preserve invariants, update protocol/extension conformance, propagate error handling, maintain actor/sendable correctness, or keep behavior coherent, but is not itself the primary vulnerable region.
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, SwiftFormat-only, comment-only, logging-only, rename-only, refactor-only, or otherwise not required to explain or fix the vulnerability.
- `vulnerable_line_ranges` must be compact inclusive ranges for the same pre-fix vulnerable region, formatted as `[[start_line, end_line], ...]`.
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant.
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region.
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain the reason in the hunk annotation.

====================
SWIFT VULNERABILITY ANALYSIS FOCUS
====================
When applicable, analyze whether the vulnerability involves one or more of:
- crash-triggered denial of service from `fatalError`, forced unwrap, forced cast, precondition failure, or out-of-bounds access
- incorrect optional handling
- invalid index or range assumptions
- integer overflow, truncation, sign conversion, or length mismatch
- parser or deserialization flaws
- file path, URL, archive, symlink, or permission handling issues
- authorization or validation bypass
- misuse of `try!`, `as!`, `unsafelyUnwrapped`, `withoutActuallyEscaping`, or `unsafeBitCast`
- unsafe pointer misuse (`UnsafePointer`, `UnsafeMutablePointer`, raw buffer pointers, unmanaged objects)
- Objective-C / C / Foundation interop contract violations
- race conditions, actor isolation violations, non-Sendable sharing, task cancellation misuse, or async reentrancy issues
- resource exhaustion, infinite loop, or unbounded allocation
- data leakage through logging, encoding, serialization, or URL handling
- ARC lifetime misuse, retain cycle side effects, or deinit-sensitive cleanup bugs when relevant
