You are generating EXACTLY ONE canonical vulnerability record from audited repository history.

Your job is to recover repository-grounded facts from the audited commit and surrounding code, not to generate a chat conversation.

You must:
1. inspect the pre-fix and post-fix repository state,
2. identify the primary vulnerable callable or executable code unit,
3. resolve all identifiers necessary to explain exploitability and the fix,
4. recover inter-procedural context and data/control-flow,
5. reconstruct the full real fix scope,
6. extract the complete pre-fix and post-fix primary callable bodies,
7. untangle vulnerability-relevant hunks from supporting or unrelated hunks,
8. output exactly one valid JSON object following the schema below.

Treat the CVE and commit as grounding metadata, not as proof of the exact bug mechanism.
Do not invent identifiers, helper functions, locations, definitions, vulnerable regions, or patch hunks.

====================
REPOSITORY ANALYSIS REQUIREMENTS
====================
- Resolve all identifiers required for exploitability or patch understanding:
  - namespaces and `use` imports
  - classes, interfaces, traits, enums, enum cases
  - top-level functions, methods, constructors, static methods
  - closures, arrow functions, generators, iterators, callables
  - properties, typed properties, static properties, constants, class constants
  - magic methods (`__construct`, `__destruct`, `__wakeup`, `__sleep`, `__serialize`, `__unserialize`, `__toString`, `__invoke`, `__call`, `__callStatic`, `__get`, `__set`, `__isset`, `__unset`, `__clone`, `__debugInfo`)
  - attributes / annotations and their arguments
  - imported/global functions, builtins, extensions, package symbols
  - autoloading paths, reflection usage, dynamic dispatch, dynamic property access, variable variables/functions where relevant
  - serializers/deserializers, hydrators, parsers, templating helpers, database adapters, stream APIs, filesystem APIs, command-execution APIs
- Recover the caller/callee chain relevant to exploitability
- Recover attacker-controlled input path where relevant
- Recover patch scope across all changed files/hunks
- If a symbol is still unresolved after exhaustive repository search, keep it in `unresolved_identifiers`
- Prefer exact repository-grounded definitions over paraphrases
- Prefer exact patch fidelity over explanatory creativity
- Do not over-resolve unrelated symbols; only include identifiers material to exploitability, control flow, data flow, taint flow, lifecycle behavior, or the fix

====================
PHP-SPECIFIC ANALYSIS REQUIREMENTS
====================
- Explicitly reason about any relevant:
  - superglobals (`$_GET`, `$_POST`, `$_REQUEST`, `$_COOKIE`, `$_FILES`, `$_SERVER`, `$_SESSION`)
  - HTTP routing, middleware, controller, request/response, CLI, queue, cron, webhook, RPC, template rendering, upload, or deserialization entrypoints
  - stream wrappers (`file://`, `php://`, `data://`, `zip://`, `phar://`, custom wrappers)
  - `include`, `require`, `include_once`, `require_once`
  - `eval`, `assert`, dynamic code loading, reflection, autoloading, Composer package resolution
  - `serialize` / `unserialize`, `json_decode`, XML/DOM/SimpleXML parsing, YAML parsing, framework model hydration, object mapping
  - weak typing / type juggling, numeric-string coercion, loose comparison, strict_types boundaries, nullable/mixed/union/intersection types
  - filesystem APIs, process execution APIs, URL fetchers, database query builders, ORM layers, template engines
  - framework-specific surfaces such as Laravel, Symfony, WordPress, Drupal, Magento, Slim, CakePHP, CodeIgniter, Laminas, Yii if relevant
- If PHP-specific constructs are not relevant, do not force them into the record.

====================
IDENTIFIER DEFINITION RULES
====================
- Every identifier with kind = "function" MUST include the complete callable definition exactly as it appears in the repository.
- Function definitions may be:
  - named functions,
  - methods,
  - static methods,
  - constructors,
  - closures,
  - arrow functions,
  - generators,
  - magic methods,
  - trait methods,
  - anonymous class methods when they are the true vulnerable executable unit.
- The `definition` field for functions MUST be valid PHP source code, not plain English.
- The `definition` field for classes, interfaces, traits, enums, properties, constants, attributes, and imports MUST be the actual code definition, not an English description.
- `vulnerable_function_pre` is a top-level attribute containing the full verbatim body of the primary vulnerable callable as it existed BEFORE the fix, including the vulnerable line(s).
- `vulnerable_function_post` is a top-level attribute containing the full verbatim body of the primary vulnerable callable as it existed AFTER the fix, showing the patched line(s).
- Both `vulnerable_function_pre` and `vulnerable_function_post` must be complete, syntactically valid PHP callable definitions extracted from the repository at `pre_state` and `post_state` respectively.
- If the primary vulnerable logic is in a closure, arrow function, trait method, magic method, anonymous class method, or dynamically assigned callable, extract that exact executable code unit and use the nearest stable callable identifier as `primary_function`.

====================
TYPE / NULLABILITY / COERCION RULES
====================
- Pay special attention to PHP-specific exploitability caused by:
  - weak typing and type juggling
  - loose comparisons (`==`, `!=`)
  - implicit scalar conversions
  - nullable types, `mixed`, unions, intersections
  - absent type declarations
  - dynamic property creation/access
  - array/object confusion
  - string-to-path / string-to-command / string-to-URL coercions
  - framework/container hydration producing objects or arrays with looser structure than expected
- If `declare(strict_types=1)` or the absence of strict types materially affects exploitability, resolve it and include it in `resolved_identifiers`.
- If taint depends on PHP trusting user-controlled strings, arrays, serialized payloads, uploaded files, or headers without normalization, state that explicitly.

====================
PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant.
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, taint flow, authz/authn check, path validation, command construction, deserialization contract, reflection/autoload boundary, type/strictness guard, stream-wrapper restriction, parser validation, lifecycle rule, or other root-cause-bearing logic.
- A hunk is `supporting_hunk` if it is required for the security fix to work correctly, preserve invariants, update dependent logic, or keep behavior coherent, but is not itself the primary vulnerable region.
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, comment-only, logging-only, refactor-only, rename-only, dead-code-only, or otherwise not required to explain or fix the vulnerability.
- `vulnerable_line_ranges` must refer only to the true pre-fix vulnerable region in the primary vulnerable file, not every touched line in the commit.
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant.
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region.
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain the reason in the hunk annotation.

- `vulnerable_function_post` must be a non-empty string containing the complete post-fix callable definition

- `patch_relevance` must be present even if one or more of its arrays are empty
- Keep `macros_resolved` in `context_quality` for cross-language schema compatibility; for PHP it should normally be 0
