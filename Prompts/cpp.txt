You are generating EXACTLY ONE canonical vulnerability record from audited repository history for a C++ codebase.

Your job is to recover repository-grounded facts from the audited commit and surrounding code, not to generate a chat conversation.

You must:
1. inspect the pre-fix and post-fix repository state,
2. identify the primary vulnerable C++ item,
3. resolve all identifiers necessary to explain exploitability and the fix,
4. recover inter-procedural context and data/control-flow,
5. reconstruct the full real fix scope,
6. extract the complete pre-fix and post-fix primary item bodies,
7. untangle vulnerability-relevant hunks from supporting or unrelated hunks,
8. output exactly one valid JSON object following the schema below.

Treat the CVE and commit as grounding metadata, not as proof of the exact bug mechanism.
Do not invent identifiers, helper functions, locations, definitions, vulnerable regions, or patch hunks.

====================
C++ IDENTIFIER DEFINITION RULES
====================
- Every identifier with kind = "function", "method", "constructor", "destructor", or "operator" MUST include the COMPLETE body (from full signature through closing brace) as the `definition` value.
- Every identifier with executable kind MUST include `function_body` and it MUST contain the COMPLETE body as valid source code.
- Every identifier with kind = "struct", "class", "union", or "enum" MUST include the COMPLETE code definition as the `definition` value.
- Every identifier with kind = "struct", "class", or "union" MUST include `type_body` and it MUST contain the COMPLETE code body, including opening and closing braces.
- The `definition` field for executable items and types MUST be valid C++ source code, not plain English.
- The `definition` field for macros, constants, typedefs, aliases, structs, classes, unions, and enums MUST be the actual code definition, not an English description.
- `vulnerable_function_pre` is a top-level attribute containing the full verbatim body of the primary vulnerable function or method as it existed BEFORE the fix.
- `vulnerable_function_post` is a top-level attribute containing the full verbatim body of the primary vulnerable function or method as it existed AFTER the fix.
- Both `vulnerable_function_pre` and `vulnerable_function_post` must be complete, syntactically valid C++ source code function or method bodies extracted from the repository at `pre_state` and `post_state` respectively.
- Every `resolved_identifiers[]` item MUST include `state` with one of `pre`, `post`, or `both`.
- Do not collapse changed helper definitions across states into one entry; emit separate `state = "pre"` and `state = "post"` entries when the code differs.
- If the primary vulnerable item is a member function, preserve the full method body and identify it using a stable qualified name such as `Namespace::Type::method`.

====================
PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant.
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, memory-safety condition, bounds check, state reset, pointer/reference validity, object lifetime, ownership transfer, RAII cleanup behavior, parser validation, arithmetic guard, authorization condition, exception safety, iterator/range validity, virtual dispatch correctness, or other root-cause-bearing logic.
- A hunk is `supporting_hunk` if it is required for the security fix to compile, link, preserve invariants, maintain lifetime/ownership correctness, update template or inheritance-related dependent logic, or propagate the fix safely, but is not itself the primary vulnerable region.
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, comment-only, logging-only, refactor-only, rename-only, dead-code-only, or otherwise not required to explain or fix the vulnerability.
- `vulnerable_line_ranges` must be compact inclusive ranges for the same pre-fix vulnerable region, formatted as `[[start_line, end_line], ...]`.
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant.
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region.
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain the reason in the hunk annotation.

====================
C++ VULNERABILITY ANALYSIS FOCUS
====================
When applicable, analyze whether the vulnerability involves one or more of:
- use-after-free, double free, invalid free
- null dereference
- out-of-bounds read/write
- integer overflow, truncation, signedness conversion, size mismatch
- parser/deserialization flaws
- iterator invalidation or container misuse
- dangling references or lifetime misuse
- ownership bugs involving raw pointers, smart pointers, move/copy semantics, or RAII
- uninitialized data use
- exception safety failures
- virtual dispatch or inheritance-related confusion
- authorization or validation bypass
- resource exhaustion, infinite loop, or unbounded allocation
- concurrency issues involving shared state, atomics, locks, or thread-unsafe objects where relevant
