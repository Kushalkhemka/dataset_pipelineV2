Your job is to recover repository-grounded facts from the audited commit and surrounding code, not to generate a chat conversation.

6. extract the complete pre-fix and post-fix primary callable body,
7. untangle vulnerability-relevant hunks from supporting or unrelated hunks,
8. output exactly one valid JSON object following the schema below.

Treat the CVE and commit as grounding metadata, not as proof of the exact bug mechanism.
Do not invent identifiers, helper symbols, locations, definitions, vulnerable regions, or patch hunks.

IDENTIFIER DEFINITION RULES
====================

- The `definition` field for functions/methods/callbacks MUST be valid JavaScript/TypeScript source code, not plain English.
- The `definition` field for imports, exports, constants, classes, object literals, enum-like constructs, and configuration objects MUST be the actual code definition, not an English description.
- `primary_symbol_pre` is a top-level attribute containing the full verbatim body of the primary vulnerable callable unit as it existed BEFORE the fix, including the vulnerable line(s).
- `primary_symbol_post` is a top-level attribute containing the full verbatim body of the primary vulnerable callable unit as it existed AFTER the fix, showing the patched line(s).
- Both `primary_symbol_pre` and `primary_symbol_post` must be complete, syntactically valid JavaScript/TypeScript callable bodies extracted from the repository at `pre_state` and `post_state` respectively.
- Every `resolved_identifiers[]` item MUST include `state` with one of `pre`, `post`, or `both`.
- Do not collapse changed helper definitions across states into one entry; emit separate `state = "pre"` and `state = "post"` entries when the code differs.
- If the vulnerable logic is implemented as an inline callback, arrow function, class method, or assigned function expression, extract that exact code form verbatim.

====================
PATCH UNTANGLING RULES
====================
- Do not assume every changed hunk in the fix commit is vulnerability-relevant.
- Separate the patch into:
  1. `security_fix_hunks`
  2. `supporting_hunks`
  3. `unrelated_hunks`
- A hunk is `security_fix_hunk` only if it directly changes the vulnerable condition, exploitability path, unsafe parser logic, unsafe deserialization logic, input validation, bounds check, state reset, permission check, path normalization, sanitizer, authorization logic, escaping/encoding logic, prototype access, regex validation, or other root-cause-bearing logic.
- A hunk is `supporting_hunk` if it is required for the security fix to preserve behavior, update dependent code, adjust imports/exports, update framework wiring, keep the program correct, or make the patch coherent, but is not itself the primary vulnerable region.
- A hunk is `unrelated_hunk` if it is test-only, formatting-only, lint-only, comment-only, logging-only, refactor-only, rename-only, dead-code-only, or otherwise not required to explain or fix the vulnerability.
- `vulnerable_line_ranges` must refer only to the true pre-fix vulnerable region in the primary vulnerable file, not every touched line in the commit.
- Preserve the full patch diff in `patch.diff`, but separately identify which hunks are vulnerability-relevant.
- If a changed file contains both relevant and irrelevant hunks, do not mark the entire file as vulnerable; isolate the specific hunk or region.
- If uncertainty remains about whether a hunk is supporting vs unrelated, prefer `supporting_hunk` and explain why.
